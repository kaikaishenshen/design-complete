# Polish 流程

发布到用户之前最终的端到端质量检查。Polish 是审计中所有发现的*解决方案*。

## 步骤 1：确定目标

解析目标（路径、URL 或表面名称）。如果未提供，从工作上下文推断。如果无法推断，询问。

最近的评审快照（来自 `critique`）充当待办事项清单。运行以下命令获取：

```bash
slug=$(node .workbuddy/skills/impeccable/scripts/critique-storage.mjs slug "<解决后的路径>")
node .workbuddy/skills/impeccable/scripts/critique-storage.mjs latest "$slug"
```

## 步骤 2：重新运行审计

```bash
node .workbuddy/skills/impeccable/scripts/detect.mjs --json [target]
```

检测器结果告诉你在 polish 之后需要验证什么。

## 步骤 3：打磨列表

对于评审快照和检测器结果中的每个项目：

### 可访问性修复
- 修复对比度问题（将颜色调整到 OKLCH 色阶的墨色端）
- 为交互元素添加焦点指示器
- 为 `<html>` 添加 `lang` 属性
- 为图片添加 `alt` 文本
- 确保表单元素有标签

### 反模式修复
- 移除侧边条纹边框 → 使用完整边框或背景色调
- 移除嵌套卡片 → 扁平化层次
- 移除眼眉标 → 使用不同的节奏
- 替换过度使用的字体 → 引人入胜的替代品

### 排版修复
- 检查行长度（最大 75ch）
- 检查字体比例（步间 ≥ 1.25）
- 修复标题字号（h1 clamp() ≤ 6rem）

### 响应式修复
- 确保布局在移动端/平板/桌面上工作
- 检查溢出文本
- 确保触摸目标 ≥ 44px

## 步骤 4：最终检查

在用户验收之前，执行完整的预发布清单：

- [ ] 每个检测器问题都已修复
- [ ] 评审快照中每个 P0/P1 问题都已解决
- [ ] 在移动设备上测试
- [ ] 在平板设备上测试
- [ ] 在桌面设备上测试
- [ ] 颜色对比度 ≥ 4.5:1 对于正文
- [ ] 没有溢出文本
- [ ] 没有嵌套卡片
- [ ] 没有 AI 模板迹象
- [ ] 动画：如果动效是 UX 的一部分，它们是否流畅且有意义？

## 步骤 5：呈现

向用户呈现修复摘要和留下的任何剩余问题。建议下一步。
