# Critique 流程

解决一个稳定的目标，运行两个独立的评估，综合一个设计评审，持有一个快照，并询问用户接下来改进什么。聊天的回复是主要交付物；快照是未来命令的存档。

## 硬性不变量

- 评估 A（设计回顾）和评估 B（检测器/浏览器证据）都是必需的。
- 评估 A 必须在检测器发现进入父综合上下文之前完成。检测器输出是确定性的，但它仍然锚定判断。
- 如果子 agent 不可用，按顺序后备：首先完成并记录评估 A，然后运行评估 B，然后综合。
- 跳过的检测器是失败的评审运行，除非 `detect.mjs` 缺失或在进行真正尝试后崩溃。

## 步骤 1：解决一个稳定的目标

目标可以是文件、目录或 URL。解决它直到它是一个稳定的可访问引用。

- 如果提供了 URL，直接使用它。
- 如果提供了文件或目录，使用它作为目标。如果是一个目录，检测整个目录（detect 支持递归）。
- 如果没有提供目标，并且用户说的是类似"评审这个页面"的话，从工作上下文推断目标。如果无法推断，询问。

## 步骤 2：运行评估 A — 设计回顾

### 2.1 用户角色选择

自动选择 2-3 个与界面类型最相关的角色。使用参考中的选择表。如果 `CLAUDE.md` 包含来自 `impeccable init` 的 `## Design Context` 章节，也从受众/品牌信息中生成 1-2 个项目特定的角色。

### 2.2 角色输出

每个角色产生：

- **第一印象**（1-2 句）
- **评分**（1-10）：整体印象
- **关键发现**（每个角色最多 3 个）
- **优先级**（P0/P1/P2/P3）

### 2.3 综合

综合角色输出成一个连贯的评审：

- 主要发现（全局问题）
- 界面层次的 UX 评分（1-10）
- 每个最高优先级的可行改进

## 步骤 3：运行评估 B — 检测器

运行 `node .workbuddy/skills/impeccable/scripts/detect.mjs --json [target]` 获取可靠的反模式数据。将命中结果分类：

- **高优先级**：对比度违规、嵌套卡片、溢出文本
- **中优先级**：AI 模板痕迹、单调排版、重叠元素
- **低优先级**：样式偏好、建议

## 步骤 4：综合

将评估 A（人类品质）和评估 B（检测器证据）合并成单一的综合评审：

- 来自双方都同意的问题 → 高信心，优先修复
- 只来自评估 A → 以经验为支持，但需要确认
- 只来自评估 B → 可操作的确定性项目

## 步骤 5：持久化

存档评审到未来命令可检索的快照中：

```bash
node .workbuddy/skills/impeccable/scripts/critique-storage.mjs slug "<解决后的路径或 URL>"
node .workbuddy/skills/impeccable/scripts/critique-storage.mjs write <slug> <body-file>
```

## 步骤 6：呈现

呈现给用户：
1. **UX 评分**：清晰可见的 1-10 分值
2. **待修复列表**：按 P0 → P3 优先级排序
3. **建议下一步**：他们应该运行哪个命令来修复它？

## 趋势

如果项目已被评审多次，通过 `critique-storage.mjs trend <slug> 5` 检查趋势线并报告分数是上升还是下降。
