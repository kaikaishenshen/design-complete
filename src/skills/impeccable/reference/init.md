# Init 流程 — 初始化项目设计上下文

第一次为项目设置。运行 `context.mjs` 后报告 `NO_PRODUCT_MD` 时触发。

## 步骤 1：明确产品

### 1.1 收集信息

通过询问识别出有足够信息来写一个 brief。收集：

- 项目是什么？（一句话）
- 谁在使用它？（目标受众、客户档案）
- 主要目的？（销售、告知、签约用户、注册、提供工具）
- 任何现有品牌指南？（颜色、字体、语气）

### 1.2 写 PRODUCT.md

在项目根目录写一个 `PRODUCT.md`，包含：

- **产品**：一句话描述。它做什么。
- **受众**：谁在使用它，处于什么场景。
- **品牌描述**：3-5 个品牌词汇。
- **注册**：品牌或产品。
- **竞争景观**：竞争产品，如果已知。

## 步骤 2：检查视觉上下文

### 2.1 设计代码检查

运行 `ls` 并检查现有 CSS 框架或自定义 CSS 文件。查找已有：

- 品牌颜色或 CSS 自定义属性（`:root` 变量）
- 字体导入或 `@font-face` 声明
- 间距模式或排版比例

如果找到，提取到 `DESIGN.md`。

### 2.2 无设计代码：调色板生成

如果项目干净（无设计代码），运行 `palette.mjs` 脚本获取种子色。在 `DESIGN.md` 中围绕它构建一个调色板。

## 步骤 3：写 DESIGN.md

在项目根目录创建 `DESIGN.md`，包含：

```markdown
# 设计系统

## 品牌种子
- 主品牌色：[从 palette.mjs 获取]
- 描述：[从 PRODUCT.md 获取]

## 排版
- [选择的展示/正文字体对]

## 调色板
- 背景、表面、墨色、强调色、柔和色
- 使用 OKLCH
```

向用户呈现以确认或调整。

## 步骤 4：推荐起始点

给出用户可下一步运行的可操作命令：

- `/impeccable craft [feature]` — 构建东西
- `/impeccable document` — 从现有代码提取设计系统
- `/impeccable audit` — 检查质量
- `/impeccable live` — 启动实时迭代模式

可选择：停止并调用 AskUserQuestion 工具进行澄清。询问用户是否希望将 PRODUCT.md 的简短摘要追加到项目根目录的 CLAUDE.md，以便 Agent 更容易引用。如果同意，在那里追加一个简短的 **Design Context** 指针部分。
