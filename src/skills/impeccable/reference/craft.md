# Craft 流程

以 impeccable 的 UX 和 UI 质量构建功能：塑形设计、确定视觉方向、构建生产级代码、在浏览器中检查和改进，直到达到高端工作室水准。

在写代码之前，你需要：已加载 PRODUCT.md、已识别注册并加载匹配的参考文件、已为当前任务确认设计方向（来自 `shape` 或用户提供）。PRODUCT.md 是项目上下文，不是任务特定简报。

将任何已批准的视觉方向（生成的模型或指定的参考）视为构图、层次、密度、氛围、标志性主题和独特视觉表现的具体契约。不要让模型取代结构、文案、可访问性或状态设计。但如果实际结果缺少已批准方向的主要成分，则实现有误。

### 关卡：不可压缩

Craft 有**多个用户关卡**，而非一个。当 harness 具备原生图像生成能力时（Codex 通过 `image_gen`），代码前的关卡顺序为：

1. **塑形简报已确认**（步骤 1）
2. **方向问题已回答**（codex.md 步骤 A）
3. **调色板已确认**（codex.md 步骤 B）
4. **一个模型方向已批准或委派**（codex.md 步骤 D）

你必须在每个关卡停下。**仅确认塑形不等于开始编码的绿灯。** 它是开始 codex.md 步骤 A 的绿灯。因为塑形简报看起来完整就压缩关卡 2 到 4，是这个流程的主导失败模式。

当 harness 缺乏原生图像生成能力时，关卡 2-4 折叠进简报本身，塑形确认直接推进到代码。

## 步骤 0：项目基础

在塑形、代码之前：弄清楚你在什么类型的项目中工作。

查看工作目录。运行 `ls`。检查：

- 现有框架：`astro.config.mjs/ts`、`next.config.js/ts`、`nuxt.config.ts`、`svelte.config.js`、`vite.config.js/ts`、`package.json` 带框架依赖、`Cargo.toml` + Leptos/Yew、`Gemfile` + Rails。**如果找到，使用它。** 不要启动并行构建，不要引入第二个框架，不要直接写入 `dist/` 或 `build/`。无论项目使用什么流水线，尊重它。
- 现有组件库或设计系统：`src/components/`、`app/components/`、`tokens.css` / `theme.ts`、`astro.config` 的 `integrations`。在添加之前，先读取已有内容。
- 现有图标集：`lucide-react`、`@phosphor-icons/react`、`@iconify/*`、`assets/icons/` 中的手写 SVG 精灵。**使用项目中已有的**；不要引入第二套。

如果目录为空（绿地项目），不要默默选择框架。通过 AskUserQuestion 工具询问用户，附带根据简报设定的合理默认值：

```text
应该基于什么构建？
  - Astro（内容驱动品牌站、落地页、市场面的默认选择）
  - SvelteKit / Next.js / Nuxt（当简报暗示应用界面或大量交互时）
  - 单文件 index.html（一次性演示、原型、或刻意无框架的实验）
```

## 步骤 1：确定可用工具

确认 agent 的可用工具集。差异很大；在假设能力之前检查实际可用性：

- **浏览器快照**（`images/screenshot` 或等价）：可用时，在构建页面后使用视觉检查。**完成整个实现后，必须拍摄高清全页快照**，并询问用户是否需要进行三次调整。
- **浏览器控制**（计算机使用、Playwright、Puppeteer）：可用时用于交互式验证。不可用时，依赖静态检查和用户反馈。
- **图像生成**（Midjourney、DALL-E 等）：可用时，通过 codex.md 流程在写出任何代码之前探索视觉方向。**这是可以开始的信号，不是必须的。**
- **实时浏览器**（`live` 子命令）：可用时用于在交付前打磨元素。详情见 [live.md](live.md)。

### 不可用工具时的行为

当 `craft` 在不可用状态运行时，在写入代码后，必须**创建一个清晰的待办事项清单**，列出用户需要自行检查的内容（浏览器兼容性、交互测试、视觉回归测试），按风险优先级排序。将这些作为问题和建议呈现。

## 步骤 2：塑形（如同 `shape` 命令）

如果 `craft` 在没有预先存在 `shape` 的情况下被调用，（在确定框架和工具之后）执行完整的 `shape.md` 流程。**这包括所有门控棋盘**。不压缩。

## 步骤 3：构建

### 3.1 输出格式

**总是产出可定向文件。** 对于静态站点，每个区域写入自己的模板/页面文件，每个组件写入自己的组件文件。不要将整个页面倾倒在一个文件或一个骨架函数中。通过文件系统构建。

### 3.2 构建过程

按此顺序构建界面，不要走捷径：

1. **先做 HTML/CSS 框架。** 页面结构、网格、排版，零功能 - 纯视觉骨架。
2. **应用品牌/设计系统。** 使用配色和品牌字体通过 tokens。
3. **填充副本，状态。** 变体、错误状态、空状态。不要使用 Lorem ipsum。
4. **迭代交互和动画。** 结束后添加。

## 步骤 4：检查交付物

交付之前，执行*最终视觉检查*：

- 每个元素是否有正确的颜色、字体、间距？
- 所有状态是否覆盖（悬停、焦点、活动、错误、空）？
- 副本是否具体、清晰、无营销腔？
- 布局是否在移动端、平板端、桌面端工作？
- 动画是否流畅而不是突兀？

## 步骤 5：最终签名

标记为已 review 并呈现给用户。列出可用的下一步操作以继续迭代。
