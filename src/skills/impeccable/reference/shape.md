# Shape 流程

在写代码之前，塑形用户界面：明确布局、内容、交互和视觉方向。其输出是概念性的（模型、文本描述、结构大纲），不是代码。这确保设计意图在执行之前已被理解和同意。

## 步骤 1：塑形简报

### 1.1 阅读简报并提问

将用户原始请求塑造成一个清晰的简报。如果简报缺失关键信息，使用 AskUserQuestion 工具询问。需要的信息：

- **页面/功能类型**：落地页、仪表盘、设置页、结账流程、搜索结果、404 页面
- **目标受众**：谁在使用这个？现有客户、首次访问者、专家用户？
- **情绪/基调**：严肃、好玩、极简、奢华、值得信赖、尖端
- **设备优先级**：桌面优先？移动端？

### 1.2 识别注册

- **品牌注册**：设计是产品。落地页、活动、作品集、品牌站点、营销页面、长文内容。
- **产品注册**：设计服务于产品。仪表盘、管理后台、工具、设置页面、应用 UI。

注册决定大局选择：品牌注册在布局上更具实验性；产品注册优先考虑效率和清晰度。

### 1.3 结构决定

定义页面级骨架：

- **布局**：英雄、功能网格、定价表、见证部分 — 各部分以什么顺序呈现？
- **导航**：顶部栏、侧边栏、没有导航？用户如何移动？
- **内容优先顺序**：首先显示什么？什么放在折叠下方？

## 步骤 2：输出

将塑形输出为简洁的结构：

```markdown
## 已塑形：[页面/功能名称]

**注册**：品牌/产品

**布局**：
- [顶部栏/侧边栏/无导航]
- 英雄部分：[图片布局（全宽、分割等）/ 标题 / CTA]
- 功能部分：[网格/标签/卡片布局] — [部分名称]
- [定价/见证/CTA/FAQ — 如适用]

**在编写之前已检查**：
- [ ] 注册选择清楚
- [ ] 视觉方向（Vibe/Layout 原型）已定
- [ ] 内容层次排序
- [ ] 移动端考虑（折叠方向）
- [ ] 交互模式（悬停、焦点、过载状态）

**未解决问题**：
- [需要解决才能推进的问题]
```

## 门控

在推进 `craft` 之前，让用户确认塑形输出。如果用户说"开始"，则继续代码。

## 原则

- **不要过度设计。** 塑形是一个框架，不是线框图低保真度。足以开始构建即可，不要超过。
- **不要计划过多。** 一个页面或功能一次。在流程的 craft 部分对细节进行迭代。
- **当简报足够清晰时不要询问。** 如果用户说了"给我一个类似 Linear 风格的着陆页"，你就有足够的上下文。如果他们没有，问。
- **将塑形视为合同。** 一旦用户同意塑形，写一个代表已商定方向的模型或代码骨架。不要事后发明全新方向。
